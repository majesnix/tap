## Layout Architecture

### Phase 3 Layout Decision

**Layout model:** Three-pane layout is preserved with the right pane repurposed.

The existing `HexPreviewPanel` right pane (`w-80`) becomes a **dual-tab panel** with two tabs:

1. **"Hex" tab** (default) — existing `HexPreviewPanel` content, unchanged
2. **"History" tab** — new `MessageHistoryPanel` component with the history log, filter controls, and Resend action

This resolves the HIST-03 hex view collision: HIST-03 (hex string of a history entry) opens in a **Dialog** when the user clicks a history row's hex icon, rather than displacing the right panel.

**Rationale:** The right pane is already the natural place for "inspect current message" content. Extending it with a tab for "inspect past messages" is consistent without adding a new layout region. The alternative (history below FormPanel) would compress the form and add vertical scrolling to an already scrollable surface.

**AMQP Properties (PUBL-04):** Opens as a shadcn **Sheet** sliding from the right edge of the window, triggered by an "AMQP Properties" button added to the PublishBar. Sheet does not displace any panel — it overlays.

**Multi-proto tabs (PROT-04):** Tab strip replaces the single File Section in the sidebar. Each tab represents one loaded `.proto` file. The Message Type selector below it filters to that file's messages.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│  Sidebar (w-72)         │  Main (flex-1)           │  Right Panel (w-80)   │
│  ─────────────────────  │  ──────────────────────  │  ──────────────────── │
│  "Proto Sender"         │  PublishBar              │  Tabs: [Hex][History] │
│  ─────────────────────  │   + [AMQP Properties]   │                       │
│  [Proto file tabs]      │    btn (Sheet trigger)   │  Active tab content:  │
│   Tab1 | Tab2 | +       │  ──────────────────────  │  - Hex: HexPreview    │
│  ─────────────────────  │  FormPanel (scrollable)  │  - History: log table │
│  Message Type selector  │                          │    + filter           │
│  ─────────────────────  │                          │                       │
│  ConnectionSection      │                          │                       │
└─────────────────────────────────────────────────────────────────────────────┘
                                            ┌ Sheet (overlay, right edge) ┐
                                            │ AMQP Properties (PUBL-04)   │
                                            └─────────────────────────────┘
```

### Proto File Tabs — Sidebar (PROT-04)

**Component:** `src/components/sidebar/FileSection.tsx` (modified in-place)

Replaces the current single-file display with a shadcn Tabs component. Each open `.proto` file is a tab.

Layout:
- `TabsList` horizontal, `w-full`, tabs truncate file names to 16 chars with tooltip for full path
- Each `TabsTrigger` shows the filename (no path), with a close `×` icon button on the right (`size="icon" variant="ghost" className="h-4 w-4"`)
- A `+` button after the last tab triggers a new file open dialog (same as existing FileSection behavior)
- Maximum visible tabs: 4 (scroll if more — use `overflow-x-auto` on `TabsList`)

Active tab determines which schema populates the Message Type selector below. Switching tabs updates `useProtoStore.setFile()` with the selected file's schema.

Tab label typography: `text-xs font-semibold` for filename (Hint size at semibold weight — exception justified by horizontal space constraint).

### WellKnownType Fallback Controls — Delta (PROT-03)

FORM-09/Phase 1 already handles Timestamp (datetime-local input) and Duration (text + regex). Phase 3 adds improved fallback controls for remaining WKTs:

The existing `WellKnownTypeField` "fallback" branch (any WKT that is not Timestamp or Duration) shows a plain text input with a badge. PROT-03 requires that **Any**, **FieldMask**, **Struct**, and scalar wrapper types (BoolValue, StringValue, Int32Value, etc.) remain as the text-based fallback — no new purpose-built controls are needed beyond what Phase 1 delivered.

**PROT-03 delta:** Only visual/UX clarification needed:
- The wkt badge text on fallback inputs changes from `wkt` to the actual type name (e.g. "Any", "FieldMask") — this is already implemented in `WellKnownTypeField` via the `{isFallback && <Badge ...>{wkt}</Badge>}` branch
- Placeholder text changes to `{wkt} (JSON)` for Any and Struct, and `{wkt} value` for wrapper types — to communicate the expected encoding format

No new shadcn components needed for PROT-03. Modification is within `WellKnownTypeField.tsx`.

## Interaction States

### History Panel Tab Auto-Switch

| Trigger | Behavior |
|---------|---------|
| Message sent successfully (toast fires) | Active tab switches from "Hex" to "History" automatically |
| Manual tab click | User can always switch back to "Hex"; no auto-switch on subsequent sends overrides a manual "Hex" selection |

Implementation note: track `userManualTab` boolean in local state; auto-switch only when `userManualTab === false`.

### History Row Selection (HIST-02)

| State | Visual | Behavior |
|-------|--------|---------|
| Default | Standard row, `bg-transparent` | No selection |
| Hover | `bg-muted/50` | Shows intent |
| Selected | `bg-muted` persisted | Row clicked — form re-populated with entry field values, active tab switches to "Hex" to show re-encoded preview |
| Deselected | Reverts to default | User edits any form field; selection cleared |

Re-populate behavior (HIST-02): clicking a history row calls `useProtoStore.setSelectedType()` with the entry's message type, then injects saved field values into the form via `reset()` from `react-hook-form`. Tab switches to "Hex" so user sees the re-encoded result before resending.

### Resend Button (HIST-02 + HIST-01)

| State | Visual | Behavior |
|-------|--------|---------|
| Idle | `variant="default"` (accent), `RotateCcw` icon | Ready |
| No active connection | `disabled`, grayed | Tooltip: "Connect to a RabbitMQ profile to resend." |
| Sending | `disabled`, `<Loader2 className="animate-spin" />` | Prevents double-submit |
| Success | Returns to idle; new history entry appended at top | Toast: "Message resent to [target]" |
| Failed | Returns to idle | Toast: "Resend failed: [error message]" |

### AMQP Properties Sheet (PUBL-04)

| State | Visual | Behavior |
|-------|--------|---------|
| Closed | "Properties" button visible in PublishBar | No overlay |
| Open | Sheet slides from right; overlay darkens main content | User can edit fields |
| Apply clicked | Sheet closes; state stores properties | Next send includes these properties |
| Reset clicked | All fields revert to defaults within the sheet | Sheet stays open |
| Sheet dismissed (click outside or Esc) | Sheet closes WITHOUT applying changes | In-progress edits discarded |

### Multi-proto Tab Switching (PROT-04)

| State | Visual | Behavior |
|-------|--------|---------|
| Single file open | Single tab active, `+` button after | Same as Phase 2 behavior |
| Second file opened | New tab appears, becomes active | Message Type selector re-populates with new file's messages |
| Tab switched | Previous tab tab label returns to default style | Message Type selector updates to selected tab's schema; form resets to new type's defaults |
| Tab closed (× icon) | Tab removed | If it was active, previous tab activates; if no tabs remain, empty state shown |
| Tab close — last tab | Tab removed | Sidebar returns to "No proto file loaded" state; form shows empty state |

### AMQP Delivery Mode Switch (PUBL-04)

| State | Visual | Notes |
|-------|--------|-------|
| Off (non-persistent) | Switch left, gray | AMQP delivery-mode = 1 |
| On (persistent) | Switch right, `bg-primary` accent | AMQP delivery-mode = 2 |

Switch label left: "Non-persistent" | Switch label right: "Persistent". Labels always visible — not toggled by switch state.
