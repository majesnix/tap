In src-tauri/src/commands/plan_runner.rs, implement the execute_step command:

Signature:
```
#[tauri::command]
pub async fn execute_step(
    profile_name: String,
    step: PlanStep,
    pool_state: State<'_, PoolState>,
    run_state: State<'_, Mutex<Option<PlanRunState>>>,
) -> Result<StepResult, String>
```

Implementation outline:

1. Acquire token BEFORE any await. Clone the CancellationToken (or create a new
   one and store it in run_state if not already set):
   ```
   let token = {
       let mut guard = run_state.lock().unwrap();
       if guard.is_none() {
           *guard = Some(PlanRunState { token: CancellationToken::new() });
       }
       guard.as_ref().unwrap().token.clone()
   };
   // Guard dropped here — before first .await
   ```

2. Clone pool_state before .await (pitfall #6 — MutexGuard is not Send):
   ```
   let pool = {
       let guard = pool_state.lock().unwrap();
       guard.as_ref().ok_or_else(|| "No active connection".to_string())?.clone()
   };
   ```

3. Encode the message. Call the existing encode module:
   - Parse step.field_values as serde_json::Value
   - Call crate::commands::encode::encode_message_from_pool(&pool, &step.message_type, form_values).await
     or use the encode command's underlying logic directly. Check encode.rs for the
     available public function signature and use whatever is exported. If encode_message
     is not public, replicate the minimal encoding inline (DescriptorPool lookup +
     DynamicMessage + encode_to_vec pattern from RESEARCH.md pitfall notes).
   - On error: return Ok(StepResult { step_id: step.id, status: "error".into(),
       reply: None, error: Some(e.to_string()) })

4. Determine routing info from step.target (exhaustive match):
   - Queue { queue } → exchange = "", routing_key = queue
   - Exchange { exchange, routing_key } → exchange, routing_key

5. Branch on step.response_mode:

   Branch A — NoWait { delay_ms }:
   - Open connection + channel from pool
   - basic_publish to (exchange, routing_key) with encoded bytes, empty AMQP props
   - tokio::time::sleep(Duration::from_millis(delay_ms)).await
   - return Ok(StepResult { step_id: step.id, status: "done".into(), reply: None, error: None })

   Branch B — CorrelationId { reply_queue, timeout_ms }:
   - Open connection + channel from pool
   - Generate correlation_id = Uuid::new_v4().to_string()
   - Set up consumer on reply_queue FIRST (basic_consume with auto_ack=false,
     consumer_tag = format!("tap-run-{}", &step.id[..8]))
   - Build AMQP props: correlation_id = ShortString::from(correlation_id.as_str()),
     reply_to = ShortString::from(reply_queue.as_str())
   - basic_publish with props + encoded bytes
   - Pin deadline OUTSIDE the reply loop:
     let deadline = tokio::time::sleep(Duration::from_millis(timeout_ms));
     tokio::pin!(deadline);
   - Reply loop (tokio::select! three branches):
     - delivery branch: check correlation_id via
       delivery.properties.correlation_id().as_ref().map(|s| s.as_str())
       against correlation_id.as_str(). If match: ack, build ReplyMessage, return done.
       If no match: nack with requeue=true, continue loop.
     - deadline branch: return error "Timeout waiting for reply"
     - token.cancelled() branch: return error "Cancelled"

   Branch C — FirstArrival { reply_queue, timeout_ms }:
   - Open connection + channel from pool
   - Set up consumer on reply_queue FIRST (basic_consume with auto_ack=false)
   - basic_publish with encoded bytes (no correlation_id, no reply_to in props)
   - Pin deadline OUTSIDE loop
   - tokio::select! three branches same as Branch B but no correlation check:
     - First delivery received: ack, build ReplyMessage, return done.
     - deadline: return error "Timeout waiting for reply"
     - token.cancelled(): return error "Cancelled"

6. ReplyMessage construction helper (inline or private fn):
   - routing_key: delivery.routing_key.to_string()
   - exchange: delivery.exchange.to_string()
   - hex_string: hex encoding of delivery.data (use itertools or format!("{:02x}", b) join)
   - decoded + decoded_as: attempt to decode delivery.data using pool's DescriptorPool
     with [step.message_type] as the single-element candidate list (option (a) from
     RESEARCH.md Open Question #4). On success: decoded = Some(serde_json::to_value(...)),
     decoded_as = Some(step.message_type.clone()). On failure: decoded = None, decoded_as = None.

7. Wire execute_step into lib.rs invoke_handler:
   Add commands::plan_runner::execute_step and commands::plan_runner::cancel_plan_run

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| JS frontend → execute_step | step.field_values is a JSON string from user form; parsed by Rust |
| RabbitMQ → reply consumer | delivery.data is untrusted broker payload; decoded with error handling |
| execute_step → pool | pool holds live AMQP connections; state must be pre-checked |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-03 | Tampering | step.field_values parsing | mitigate | serde_json::from_str returns Err on malformed JSON; propagated as StepResult error, not panic |
| T-22-04 | Denial of Service | reply loop — non-matching correlation IDs | mitigate | Non-matching deliveries NACKed with requeue=true; loop terminates on deadline or cancellation — never infinite (pitfall #60) |
| T-22-05 | Elevation of Privilege | correlation_id read from headers | mitigate | correlation_id read from AMQP properties ONLY (pitfall #58); header injection cannot spoof it |
| T-22-06 | Denial of Service | deadline pin inside loop | mitigate | tokio::pin! outside loop prevents per-iteration reset; 10s absolute timeout enforced (pitfall #4) |
| T-22-SC | Tampering | Cargo.toml uuid addition | mitigate | uuid crate pre-approved in RESEARCH.md Package Legitimacy Audit (uuid-rs/uuid, verified); human checkpoint not needed for pre-audited crate |
</threat_model>
