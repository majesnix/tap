## Color

Source: `src/index.css` oklch neutral palette + `PublishBar.tsx` semantic status badge patterns (established in-codebase convention).

This app uses a **neutral-monochrome base with semantic-only chromatic status feedback**. The 60/30/10 framework maps as follows:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--background)` ‚Äî oklch(1 0 0) light / oklch(0.145 0 0) dark | All surfaces: PlanRunBar background, step list background, editor panel |
| Secondary (30%) | `var(--card)` / `var(--muted)` ‚Äî oklch(0.97 0 0) light / oklch(0.269 0 0) dark | Step row hover state, disabled input backgrounds |
| Accent ‚Äî in-progress/warning | amber-500 tint ‚Äî `bg-amber-500/10 text-amber-700 dark:text-amber-400 border-amber-500/20` | Sending badge, WaitingResponse badge |
| Accent ‚Äî success | emerald-500 tint ‚Äî `bg-emerald-500/10 text-emerald-700 dark:text-emerald-400 border-emerald-500/20` | Done badge, run summary "‚úì N/M succeeded" text |
| Destructive | `var(--destructive)` ‚Äî oklch(0.577 0.245 27.325) light / oklch(0.704 0.191 22.216) dark | Stop button, Error badge, run summary "‚úó N/M succeeded" text |

Accent reserved for: **status feedback only** ‚Äî emerald (Done step, success summary), amber (Sending/WaitingResponse step). No decorative accent surfaces anywhere in this phase.

Active step row highlight: `bg-accent text-accent-foreground` ‚Äî same token as Phase 21 selection highlight (D-10). This is the shadcn `--accent` neutral tint, not the chromatic amber/emerald palette. Source: `StepListPanel.tsx` line 137.

## Interaction States

### PlanRunBar button states

| State | Component | Visual |
|-------|-----------|--------|
| Idle, runnable | `Button variant="default"` | Play icon (Lucide `Play`, 14px) + "Run" label |
| Running | `Button variant="destructive"` | Square icon (Lucide `Square`, 14px) + "Stop" label |
| Disabled ‚Äî no steps | `Button variant="default" disabled` | "Run" label, grayed. Wrapped in `Tooltip`: "Add steps to run" |
| Disabled ‚Äî no profile | `Button variant="default" disabled` | "Run" label, grayed. Wrapped in `Tooltip`: "Select a connection profile first" |
| After completion | Replaced by run summary + "Re-run" button | See Run Summary section |

Disabled button pattern: wrap button in `<span>` inside `<Tooltip>` ‚Äî matches `PublishBar.tsx` TooltipProvider pattern.

### Stop-on-error toggle

Component: shadcn `Switch` + `Label` (already in `/src/components/ui/switch.tsx`).
Label text: "Stop on error" ‚Äî placed left of the switch.
Layout: left of Run/Stop button in PlanRunBar. When toggled, persists immediately via `usePlanStore.updatePlan`.

### Run summary (post-completion)

Replaces the Run button area inline in PlanRunBar. Layout: `[summary text] [Re-run button]`.
- Success: `"‚úì {N}/{M} succeeded"` ‚Äî text-emerald-700 dark:text-emerald-400, font-semibold, text-sm
- Partial or full failure: `"‚úó {N}/{M} succeeded"` ‚Äî text-destructive, font-semibold, text-sm
- Re-run: `Button variant="outline" size="sm"` ‚Äî label "Re-run". Clicking resets all step statuses to Pending and starts a new run. Summary clears as soon as a new run starts.

### Step status badges

Placed inline in each step row (StepListPanel), right-aligned, using `Badge` component with `className` overrides (not new variants ‚Äî matches PublishBar outcome badge pattern):

| Status | Badge variant | className override |
|--------|---------------|--------------------|
| Pending | `outline` | default outline ‚Äî neutral, no fill |
| Sending | `outline` | `bg-amber-500/10 text-amber-700 dark:text-amber-400 border-amber-500/20` |
| WaitingResponse | `outline` | `bg-amber-500/10 text-amber-700 dark:text-amber-400 border-amber-500/20` + `Loader2 animate-spin` icon (14px) prepended |
| Done | `outline` | `bg-emerald-500/10 text-emerald-700 dark:text-emerald-400 border-emerald-500/20` |
| Error | `outline` | `bg-destructive/10 text-destructive border-destructive/20` |

Badge text labels: "Pending", "Sending", "Waiting‚Ä¶", "Done", "Error".

### Active step highlight in StepListPanel

Actively-executing step (status: Sending or WaitingResponse): `bg-accent text-accent-foreground` ‚Äî identical to selection highlight. Source: D-10, StepListPanel.tsx existing pattern. Auto-scroll: `scrollIntoView({ behavior: 'smooth', block: 'nearest' })`.

### StepFieldEditor read-only during run

All inputs in StepFieldEditor receive `disabled` prop while `usePlanExecutionStore.runningPlanId !== null`. No visual indicator beyond the standard disabled state (shadcn grayed input). StepFieldEditor auto-switches to display the active step's fields during a run (D-10).

### Per-step error toast (stop_on_error = false)

When a step fails and the run continues, show a Sonner toast:
- Type: `toast.error(...)` ‚Äî destructive style, 5 seconds.
- Content: `"Step '{step.name}' failed: {error}"`.
- Source: `sonner` package already installed (CONTEXT.md code_context).

## PlanRunBar Layout

The sticky header bar spans the full width at the top of `PlanDetailPanel`, above the step list + editor split. Mirrors `PublishBar` placement at the bottom of the main layout.

Layout (left to right, single row, `flex items-center gap-4`):
```
[Plan name ‚Äî truncated, font-semibold text-sm flex-1] | [Stop on error switch + label] | [Run/Stop button OR Run summary + Re-run]
```

Container: `flex items-center gap-4 bg-card border-b border-border px-4 py-2 shrink-0`

Plan name: `<h2 className="text-sm font-semibold truncate flex-1">` ‚Äî read-only, non-editable in this bar (authoring is in StepFieldEditor header if needed).
