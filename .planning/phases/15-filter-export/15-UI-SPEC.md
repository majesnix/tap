## Layout Contract

### Filter Row Placement

The filter row is a **new sibling row** in `MessageFeedTab.tsx`. It renders unconditionally regardless of mode (Drain or Subscribe). Exact position in the render tree:

1. Mode toggle row (existing)
2. `ResponseQueuePicker` (existing)
3. SubscribePanel slot — conditional on `mode === "subscribe"` (existing)
4. **Filter row — NEW, always rendered** ← Phase 15 insertion point
5. Feed header (`bg-muted` bar with count + Clear button) (existing)
6. Feed body / ScrollArea (existing)

### Filter Row Layout

```
| [ routing key input — flex-1 ] [ Content-type ▾ ] [ ↓ Export ] |
```

- Container: `px-4 py-2 border-b border-border flex items-center gap-2`
- Routing key input: `flex-1` — expands to fill remaining width
- Content-type Select: fixed width, enough for longest content-type label (`w-52` or similar)
- Export button: `variant="outline"` with `Download` icon (`size="sm"` to match other toolbar buttons)

### Feed Header Count Label

- **When no filter active:** `"42 messages"` / `"1 message"` / `"No messages"` (existing logic)
- **When any filter active:** `"12 of 42 messages"` (visible count of total count)
- A filter is considered active when: routing key input is non-empty OR content-type is not `"all"`

## Interaction Contract

### Routing Key Filter (FILT-01)

- Input type: text, placeholder `Filter by routing key`
- Match mode: case-insensitive `includes()` substring match against `FeedMessage.routingKey`
- Reactivity: immediate on each keystroke (no debounce needed — purely derived state, no IPC)
- Clear: user deletes text; no separate clear button required

### Content-Type Filter (FILT-02)

- Control: shadcn `Select` dropdown
- Options: dynamically derived from distinct `contentType` values in `messages[]`, sorted alphabetically, always prepended with an `"All"` option
- Default selected: `"All"` (shows all messages)
- Null/empty `contentType` appears as `"(none)"` in the dropdown label and filters by `null` internally
- Reactivity: immediate on selection change

### Combined Filter Behavior

- When both filters are active: AND logic — visible set requires routing key substring match AND content-type match simultaneously (D-05)
- `visibleMessages` is computed as `useMemo` derived from `messages`, `filterRoutingKey`, and `filterContentType`

### Export Button (XPRT-01)

- Label: `Export` with `Download` icon to the left
- `variant="outline"`, `size="sm"` matching existing toolbar buttons
- **Disabled when:** `visibleMessages.length === 0`
- On click: generates default filename `feed-export-{ISO-timestamp}.json` (colons in timestamp replaced with dashes), opens native Tauri save dialog via `@tauri-apps/plugin-dialog` `save()`
- If user cancels dialog: silent — no toast, no state change (D-08)
- On successful write: `toast.success("Exported {N} messages")` via existing `sonner` wiring (D-13)
- On write failure: `toast.error("Export failed: {reason}")` — surface the error from `writeTextFile`

### Empty States

| Condition | Display |
|-----------|---------|
| No messages in feed | `"Select a queue and choose a mode"` (existing, in ScrollArea body) |
| Messages in feed, but none match filter | `"No messages match filter"` (new, shown in ScrollArea body in place of accordion) |

## Export JSON Shape

Per D-10 through D-12:

```json
{
  "exportedAt": "<ISO 8601 string>",
  "messageCount": 42,
  "messages": [
    {
      "routingKey": "order.created",
      "exchange": "orders",
      "contentType": "application/octet-stream",
      "timestamp": "2026-05-21T14:31:58.000Z",
      "decodedAs": "OrderEvent",
      "decoded": { "orderId": "abc-123", "amount": 99.99 },
      "error": null
    }
  ]
}
```

Fields omitted from export: `id` (internal UUID), `hexString` (raw bytes, noise in JSON export).
`timestamp` serialized as ISO 8601 string from epoch seconds (`new Date(ts * 1000).toISOString()`); `null` if publisher did not set it.
