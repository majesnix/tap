## Spacing Scale

Declared values — all multiples of 4. Scale is density-tuned for a form-heavy desktop tool,
not a marketing page. Tighter inner spacing preserves vertical real estate in nested forms.

| Token | Value | Usage                                                          |
|-------|-------|----------------------------------------------------------------|
| xs    | 4px   | Icon-to-label gap, inline checkbox gap, radio button gap       |
| sm    | 8px   | Input internal padding-x, label-to-input gap, badge padding, form row gap (between adjacent fields in a row) |
| md    | 16px  | Section gap within a form group, sidebar item gap              |
| lg    | 24px  | Panel internal padding, between top-level form sections        |
| xl    | 32px  | Between sidebar sections, between form and preview strip       |
| 2xl   | 48px  | Not used in Phase 1 (reserved for page-level breaks in later phases) |

Exceptions:
- Sidebar width: 240px fixed (D-02, CONTEXT.md)
- Depth-cap collapse placeholder: 8px vertical padding (compact, communicates non-interactivity)
- Bottom preview strip: 16px padding all sides; monospace code block uses 8px padding-x, 8px padding-y

Source: D-02, general form-density guidance.

## Typography

| Role    | Size  | Weight      | Line Height | Font stack     |
|---------|-------|-------------|-------------|----------------|
| Body    | 14px  | 400 regular | 1.5         | Inter          |
| Label   | 13px  | 600 semibold| 1.4         | Inter          |
| Heading | 16px  | 600 semibold| 1.3         | Inter          |
| Code    | 13px  | 400 regular | 1.6         | JetBrains Mono |

Notes:
- 2 weights only: 400 regular (Body, Code) and 600 semibold (Label, Heading).
- 14px body is correct for a Tauri desktop app at 1x display scale; marketing-tuned 16px body wastes form real estate.
- Labels at 13px/600 provide visual weight that anchors each form field without needing a third weight.
- Code/monospace row is mandatory: hex preview (D-06) renders `0a 05 68 65 6c 6c 6f` — proportional fonts make hex strings unreadable.

## Color

This is a developer tool. Use shadcn/ui zinc-based dark palette as the default. Defer to the
shadcn preset that the executor selects — color values below are semantic roles only.
Explicit hex values set at init time via the shadcn preset string.

| Role             | Shadcn semantic token | Usage                                           |
|------------------|-----------------------|-------------------------------------------------|
| Dominant (60%)   | `background`          | App window background, main panel background    |
| Secondary (30%)  | `card` / `muted`      | Sidebar background, include-path dialog bg, sub-form nested panel |
| Accent (10%)     | `primary`             | See reserved list below                         |
| Destructive      | `destructive`         | Remove-item button on repeated fields only      |

Accent (`primary`) reserved for exactly:
1. "Open .proto file" CTA button (sidebar)
2. Active/selected state ring on the message-type dropdown selection
3. Focused input field ring (focus-visible outline)

Nothing else uses `primary`. All other interactive states (hover, disabled) use `muted` or
`secondary` tokens.

Source: CONTEXT.md layout decisions, standard shadcn token system.

## Layout Contract

Source: CONTEXT.md D-01 through D-12.

### Window Shell

```
┌──────────────────────────────────────────────────────────────┐
│  240px sidebar (fixed)   │  Remaining width — main panel     │
│                          │                                    │
│  [App name / logo]       │  ┌────────────────────────────┐  │
│  ─────────────────       │  │  Dynamic proto form        │  │
│  [Proto file name]       │  │  (scrollable)              │  │
│  [Message type ▼]        │  │                            │  │
│                          │  │                            │  │
│  ─────────────────       │  └────────────────────────────┘  │
│  [Connection panel]      │                                    │
│  (placeholder, Phase 2)  │  ┌────────────────────────────┐  │
│                          │  │  Hex preview strip         │  │
└──────────────────────────┴──┴────────────────────────────┴──┘
```

- Sidebar: 240px fixed, no resize handle (D-02)
- Sidebar Phase 1 content: app title, proto file name (truncated with ellipsis if long), message type dropdown, "Open .proto file" button (D-03)
- Connection panel area: visible in sidebar as inactive placeholder labelled "RabbitMQ — available in Phase 2" with muted text, no interaction (D-03, CONTEXT.md specifics)
- Main panel: scrollable form area
- Bottom strip: collapsible hex preview, **expanded by default** when a proto file is loaded (D-07)

### Include Path Dialog

- Appears after the native file picker closes, before proto parsing begins (D-08)
- Modal dialog (shadcn Dialog component)
- Pre-populated with proto file's parent directory as first entry (D-08)
- User can add rows (each row: path display + remove button) and confirm or cancel
- On cancel: proto loading aborted; sidebar reverts to no-file state
- Paths persisted per absolute file path via tauri-plugin-store (D-09)

### Form Area

- Nested message fields: expandable inline collapsible (shadcn Collapsible), indented 16px per level
- Depth cap at level 5: render a `Collapsible` placeholder with copy "Nesting limit reached (5 levels). Expand in code." (non-interactive, muted style)
- Repeated fields: each item renders as a card-like row with a remove button; "Add item" button below the list
- oneof fields: RadioGroup renders branch choices; selecting a branch unmounts all sibling field inputs (D-14)
- Enum fields: Select component showing value name strings (not integers)
- WellKnownType Timestamp: DateTimePicker control (date + time inputs, ISO 8601 output to backend)
- WellKnownType Duration: text Input with placeholder `e.g. 1h30m` validated against duration string pattern
- Remaining WellKnownTypes: deferred to Phase 3 (PROT-03); render as raw string input with type annotation label

## Interaction Contracts

| Interaction                        | Contract                                                                                                |
|------------------------------------|---------------------------------------------------------------------------------------------------------|
| "Open .proto file" click           | Native file picker (Tauri dialog plugin) → include-path dialog → proto parse → form render             |
| Proto file loaded successfully     | Sidebar shows filename, message type dropdown populates, form renders with zero-value defaults, hex preview strip expands |
| Message type switch                | Form clears immediately, zero-value defaults load for new type (D-12, no confirmation)                  |
| Form field change                  | react-hook-form onChange, zod validation, hex re-encode after 200ms debounce (D-05)                     |
| Repeated field "Add item"          | Appends new item with zero-value defaults at end of list                                                |
| Repeated field remove              | Removes item immediately, no confirmation                                                               |
| oneof branch selection             | Selects radio, unmounts all sibling branch fields, mounts selected branch with zero-value defaults      |
| Nested message expand/collapse     | Collapsible toggle; state is local to the component (not persisted)                                     |
| Hex preview collapse/expand        | Collapsible toggle at bottom of main panel; state is local (not persisted across file loads)            |
| Include path dialog — cancel       | Aborts proto load; sidebar stays in previous state                                                      |
| Include path dialog — confirm      | Persists paths via tauri-plugin-store; triggers proto parse                                             |

## WellKnownType Note

remaining WellKnownTypes are out of scope for Phase 1 — they fall under PROT-03 (Phase 3).

input with a label annotation of the WellKnownType name. Do not block on WellKnownType
completeness in Phase 1.
