<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → repo | Workflow file defines what runs; any change to workflow YAML takes effect on next trigger |
| release.yml → GITHUB_TOKEN | Token grants write access; scope must stay minimal — build jobs should not request write permissions |
| Entitlements.plist → macOS codesign | Plist is read verbatim by Tauri bundler and passed to codesign; over-broad entitlements persist into signed binary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-01 | Tampering (supply chain) | Floating action tags in release.yml (`@v6`, `@v0`, `@v2`) | accept | Dev tool with no untrusted contributors; floating major tags are standard for this repo's ci.yml; SHA pinning adds maintenance burden without proportionate benefit |
| T-16-02 | Information Disclosure | `GITHUB_TOKEN` scope in build jobs | mitigate | Confirm `GITHUB_TOKEN` with `contents: write` permission is scoped only to the `create-release` job (already gated by `if: startsWith(github.ref, 'refs/tags/')`); build jobs omit `permissions:` block so they inherit default read-only scope |
| T-16-03 | Elevation of Privilege | Entitlements over-grant (old `temporary-exception.files.absolute-path.read-write /`) | mitigate | Remove the `temporary-exception` key in Task 2; replace with scoped `cs.*` Hardened Runtime keys only; `app-sandbox=false` is a deliberate and documented decision (dev tool must read arbitrary .proto files) |
| T-16-04 | Information Disclosure | Signing secrets exposed on workflow_dispatch (Phase 17 forward) | mitigate | The `# Phase 17: signing steps go here, guarded by: if: github.event_name == 'push'` comment is scaffolded in Task 1; actual secrets are referenced only under that conditional when Phase 17 adds them — not in Phase 16 |
</threat_model>
