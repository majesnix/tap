    ```typescript
    function handleDragOver(e: React.DragEvent<HTMLDivElement>) {
      e.preventDefault();  // REQUIRED — without this, onDrop never fires (Pitfall 1)
      setIsDraggingOver(true);
    }

    function handleDragLeave(e: React.DragEvent<HTMLDivElement>) {
      // Guard against child-element flicker (Pitfall 2): dragLeave fires when pointer
      // moves into a child element. Only clear the ring when pointer truly leaves the container.
      if (!e.currentTarget.contains(e.relatedTarget as Node)) {
        setIsDraggingOver(false);
      }
    }

    function handleDrop(e: React.DragEvent<HTMLDivElement>) {
      e.preventDefault();
      setIsDraggingOver(false);

      const blockId = e.dataTransfer.getData('blockId');
      if (!blockId || !applyBlockRef.current) return;

      const block = useBlockStore.getState().blocks.find(b => b.id === blockId);
      if (!block) return;

      let blockValues: Record<string, unknown>;
      try {
        blockValues = JSON.parse(block.content) as Record<string, unknown>;
      } catch {
        return; // silent no-op — unreachable (Save validated JSON at write time)
      }

      const skipped = applyBlockRef.current(blockValues);
      if (skipped.length > 0) {
        const n = skipped.length;
        const label = n === 1 ? 'field' : 'fields';
        toast.warning(`${n} ${label} from block not in form: ${skipped.join(', ')}`);
      }
    }
    ```

    **IMPORTANT — Toast copy:** The exact wording is `"N field from block not in form: ..."` (singular) and `"N fields from block not in form: ..."` (plural). There is NO `(s)` suffix. Match the FormPanel.tsx:158 pattern — `unknownKeys.length === 1 ? "field" : "fields"`.

    **Interpretation note (D-06 deviation):** D-06 specifies the wording as `"N field(s) from block not in form: …"` using the `(s)` template notation. This plan deliberately uses proper grammatical pluralization (`field` / `fields`) instead of the literal `(s)` suffix. Rationale: `(s)` is copywriting shorthand for "singular or plural", not an output literal — producing "1 field(s) from block not in form: ghost" in the UI would be grammatically wrong. The established codebase pattern in FormPanel.tsx (line ~158) uses `unknownKeys.length === 1 ? "field" : "fields"` for the identical toast scenario. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dataTransfer.getData → blockId → store lookup | Block ID arrives from the browser DnD payload. It is used only as a lookup key against the in-memory Zustand store — no eval, no DOM injection. |
| block.content (JSON string) → JSON.parse → setValue | Block content was validated as a JSON object at Phase 11 save time. Parsing is wrapped in try/catch. Values go to RHF setValue with type coercion. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-02-01 | Tampering | onDrop / dataTransfer.getData | accept | blockId is used as a Zustand lookup key only. An attacker injecting an arbitrary blockId gets a `blocks.find()` miss → silent return. No damage possible. |
| T-12-02-02 | Tampering | JSON.parse(block.content) | mitigate | Wrapped in try/catch with silent no-op. Block content was JSON-validated at save time (Phase 11). Values go to RHF setValue — no eval, no innerHTML. |
| T-12-02-03 | Spoofing | DnD cross-origin drag | accept | Desktop-only Tauri app. DnD from external sources (browser→app) would have no `blockId` in dataTransfer → silent return in onDrop. No cross-origin script execution. |
| T-12-02-04 | Information Disclosure | toast.warning skipped fields | accept | Skipped field names are schema field names from the user's own .proto file. No PII or secrets exposed. Dev tool context. |
</threat_model>
