# Milestone v1.7: Block Apply Completeness + History Search

**Status:** ✅ SHIPPED 2026-05-25
**Phases:** 24–26
**Total Plans:** 6

## Overview

v1.7 closed the remaining gap in block apply for complex field types (oneof / WellKnownType / map) and added full-text search to the history panel. The milestone introduced a clean two-phase plan/commit architecture for block apply that separated conflict detection from form writes — enabling the batched conflict dialog without duplicating logic across phases.

## Milestone Summary

**Key Decisions:**

- `{ buildPlan, commitApply }` two-phase `ApplyBlockRef` (Phase 25): single-function ref couldn't return plan data needed to compute `skipped` in FormPanel — clean separation allows conflict prompting
- `mapReplaceRegistry` useRef pattern (Phase 25): ProtoFormRenderer switch is frozen; `useFieldArray.replace()` reached without touching the switch via MapField registration
- Block-filled map treated as dirty after first fill (Phase 25): RHF 7.76.1 limitation accepted — block-applied map is "user-owned" after first fill; Phase 26 conflict prompt handles re-drag
- `ConflictItem` discriminated union (Phase 26): IN-01 code review finding — proper kind-tag guards prevent runtime field access on wrong conflict shape
- Conflict rows default to skip (Phase 26): Pitfall E — prevents accidental data loss if user clicks Apply without reviewing each row
- Pitfall D fix (`shouldDirty: false`) applied globally (Phase 26): all `setValue` calls in block apply now use `{ shouldDirty: false }` — had only been applied to Phase A in Phase 25

**Issues Resolved:**

- WR-01 (Phase 26): Whole-field dirty guard incorrectly rejected block fills for oneof container — corrected to check `dirtyFields[key]` boolean (not the container)
- WR-02 (Phase 26): `isJsonMode` useState called after `useDndMonitor` hook — hooks-order violation causing React warning; moved above
- CR-01 (Phase 26): zero-collision map path was not appending existing rows — append + non-colliding block rows now merged correctly
- IN-01 (Phase 26): `ConflictItem` was a union of object shapes without a kind discriminant — refactored to proper discriminated union

**Tech Debt:**

- `collectFieldNames` dead export (`historyHelpers.ts`): exported but unused at runtime; live code uses `collectSearchTokens`. Not a blocker.
- `replace()` marks dirty (Phase 25): RHF 7.76.1 limitation; accepted behavior.
- 15 human runtime verification scenarios across 3 phases: all UAT-verified by developer; code analysis cannot substitute for runtime verification of dialog lifecycle and RHF field-array state.

**Issues Deferred:**

- Block apply in JSON mode (`BLK-EXT-FUTURE-01`): currently guarded off; complex field type support deferred
- Recursive nested-message merge from a block (`BLK-EXT-FUTURE-02`): requires schema-aware deep merge
- Search field values (`HIST-FT-FUTURE-01`): decoded scalar data search deferred; requires stripping RHF internals before indexing
