<objective>
Fix the broken debounce gate in FormPanel.tsx so that encode calls are gated behind a 200ms idle
window instead of firing on every form change.

**Gap being closed:** VERIFICATION.md Truth #3 failed — `debouncedValues` is computed via
`useDebounce(latestValues.current, 200)` where `latestValues` is a React ref. Mutating
`latestValues.current` does NOT trigger re-renders, so `useDebounce` always receives the same
stale value and the debounce gate is dead. `encodeMessage` is called directly inside
`handleValuesChange` on every change with no throttling. The dead variable is suppressed with
`void debouncedValues`.

**Fix strategy (Path B — state-based, minimal blast radius):**
Replace `useRef<unknown>(null)` with `useState<unknown>(null)`. Setting state triggers a
re-render, making `useDebounce(latestValues, 200)` see the new value and start the timer.
Move `encodeMessage` out of `handleValuesChange` and into a `useEffect` that runs when
`debouncedValues` changes. Keep `ProtoFormRenderer`'s `onValuesChange` contract unchanged
(the FINAL decision remains intact).

**Why not useWatch in FormPanel:** `useWatch` requires a `control` object from `useForm`, which
lives inside `ProtoFormRenderer`. Lifting the form into `FormPanel` would violate the FINAL
decision from STATE.md ("ProtoFormRenderer is FINAL — Wave 2 replaces stub field components only,
not the renderer"). The `useState` approach fixes the exact same root cause (non-reactive debounce
input) with zero interface changes.

Purpose: Satisfy VERIFICATION.md Truth #3 and prevent unbounded IPC calls on each keystroke.
Output: Fixed FormPanel.tsx + test in src/components/form/__tests__/FormPanel.test.tsx
</objective>
