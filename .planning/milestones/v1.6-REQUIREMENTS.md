## v1 Requirements (all delivered)

### Plan Library

- [x] **PLAN-01**: User can create a named plan in the dedicated plan library view — Phase 19 ✓
- [x] **PLAN-02**: User can rename a plan — Phase 19 ✓
- [x] **PLAN-03**: User can delete a plan (with confirmation dialog) — Phase 19 ✓
- [x] **PLAN-04**: User can duplicate a plan with all its steps — Phase 19 ✓
- [x] **PLAN-05**: Plans and their steps persist across app restarts — Phase 19 ✓
- [x] **PLAN-06**: User accesses the plan library via a dedicated full-screen view (separate from the main form + tabs layout) — Phase 20 ✓

### Step Authoring

- [x] **STEP-01**: User can add a step to a plan by selecting a .proto file + message type, filling field values, choosing a target queue/exchange + routing key, and setting a response mode (no-wait / correlationId / first-arrival) — Phase 21 ✓
- [x] **STEP-02**: User can import a step from message history — field values pre-filled from a selected past send — Phase 21 ✓
- [x] **STEP-03**: User can import a step from the block library — field values pre-filled from a selected saved block — Phase 21 ✓
- [x] **STEP-04**: User can duplicate an existing step within the same plan — Phase 21 ✓
- [x] **STEP-05**: User can reorder steps in a plan via drag-and-drop — Phase 21 ✓
- [x] **STEP-06**: User can rename or delete individual steps — Phase 21 ✓

### Execution Engine

- [x] **RUN-01**: User can run a plan sequentially — steps execute one at a time in order — Phase 22 ✓
- [x] **RUN-02**: User can stop a running plan at any time — Phase 22 ✓
- [x] **RUN-03**: Each step shows a status badge throughout execution (Pending / Sending / WaitingResponse / Done / Error) — Phase 22 ✓
- [x] **RUN-04**: User can configure a per-plan setting: stop on first error, or continue to remaining steps — Phase 22 ✓
- [x] **RUN-05**: A run summary is shown on completion showing how many steps succeeded and how many failed — Phase 22 ✓
- [x] **RUN-06**: No-wait steps advance after a configurable per-step delay (default 200 ms, 0 = immediate) — Phase 22 ✓

### Response Handling

- [x] **RESP-01**: No-wait steps fire the publish and advance after the configured delay — no reply consumer opened — Phase 22 ✓
- [x] **RESP-02**: CorrelationId steps publish with a generated UUID `correlation_id` + `reply_to` AMQP property, wait for a matching reply on the configured reply queue (configurable timeout per step, default 10 s); step gets Error status on timeout — Phase 22 ✓
- [x] **RESP-03**: First-arrival steps open a consumer on a specified reply queue and accept the first message that arrives (configurable timeout per step, default 10 s); step gets Error status on timeout — Phase 22 ✓
- [x] **RESP-04**: Steps that receive a reply show the decoded protobuf response inline below the step (same collapsible key-value tree as the existing Response tab) — Phase 23 ✓
- [x] **RESP-05**: A shared scrollable feed shows all messages arriving on watched reply queues during the plan run, in chronological order — Phase 23 ✓

## Out of Scope

- **Pre/post-step scripts** (Postman-style `pm.test()` / `pm.environment.set()`): REQUEST SCRIPTING is explicitly Out of Scope in PROJECT.md. Sandboxed JS runtime is a different product.
- **Direct reply-to (`amq.rabbitmq.reply-to`)**: at-most-once delivery, one consumer per channel — use named exclusive queues for correlationId and first-arrival modes instead.
- **Plan execution scheduling** (run plan on cron / time trigger): distinct product scope.
- **Non-protobuf message types** (raw JSON, Avro): Out of Scope for v1 per PROJECT.md.
