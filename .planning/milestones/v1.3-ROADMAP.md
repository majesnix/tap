# Milestone v1.3: Publishing UX + Message Blocks

**Status:** ✅ SHIPPED 2026-05-20
**Phases:** 9–12 (4 phases)
**Total Plans:** 11
**Requirements:** 16/16

## Overview

v1.3 extended Tap in two directions: smarter publishing UX (routing key autocomplete from live exchange bindings, and per-send delivery outcome badges) and a reusable message block library (create/edit/delete/persist named JSON blocks, drag-and-drop to fill form fields). All 16 requirements delivered across 4 phases in 2 days.

## Phases

### Phase 9: Routing Key Autocomplete

**Goal**: Users get live routing key suggestions from RabbitMQ exchange bindings when targeting an exchange, replacing the blank free-text input
**Depends on**: Phase 8 (foundation complete)
**Requirements**: PUBL-01, PUBL-02, PUBL-03, PUBL-04
**Plans**: 3 plans

Plans:
- [x] 09-01-PLAN.md — Rust backend: ExchangeSummary struct + fetch_exchanges update + fetch_bindings command + lib.rs registration
- [x] 09-02-PLAN.md — Frontend contracts: install Command component, ExchangeSummary type, updated ipc.ts/store, RoutingKeyCombobox component + tests
- [x] 09-03-PLAN.md — PublishBar integration: exchange type badges, bindings useEffect, conditional combobox/input, hint text, updated tests

**Success Criteria — all passed:**
1. User selects an exchange and sees a populated suggestions list in the routing key input drawn from live RabbitMQ bindings ✅
2. User selects a `headers` or `fanout` exchange and the routing key input shows no suggestions (autocomplete suppressed) ✅
3. User sees topic exchange wildcard patterns displayed in the suggestions list with visible exchange type badge ✅
4. User has no active connection or the Management API is unreachable and the routing key input falls back to plain free-text entry with no error state shown ✅

**Key Decisions:**
- percent-encode both vhost and resource name before URL construction (NON_ALPHANUMERIC) — RabbitMQ Management API requires encoded path segments
- sort + dedup pattern for deduplicated string collections from RabbitMQ binding API
- ExchangeSummary public output struct for Management API data crossing IPC boundary
- Stale-request guard via AbortController in bindings useEffect — prevents race on rapid exchange switching
- fanout and headers exchange types suppress the combobox entirely (D-10 silent fallback)

### Phase 10: Publisher Confirms Badge

**Goal**: Users receive an explicit per-send delivery outcome from the broker — ACK, Returned, NACK, or Timeout — displayed as an ephemeral badge in the publish bar
**Depends on**: Phase 9
**Requirements**: PUBL-05, PUBL-06, PUBL-07, PUBL-08
**Plans**: 2 plans

Plans:
**Wave 1**
- [x] 10-01-PLAN.md — Rust backend: PublishOutcome struct, mandatory=true, tokio timeout, Confirmation match

**Wave 2** *(blocked on Wave 1 completion)*
- [x] 10-02-PLAN.md — Frontend: PublishOutcome type, ipc.ts return type, badge state + JSX + tests in PublishBar

**Success Criteria — all passed:**
1. User sends a message that the broker confirms delivered and sees a green ACK badge; badge disappears automatically after 3 seconds ✅
2. User sends a message with no matching binding (mandatory=true, unroutable) and sees an amber Returned badge; badge disappears after 5 seconds ✅
3. User sends a message the broker negatively acknowledges and sees a red NACK badge; badge disappears after 5 seconds ✅
4. User sends a message and broker confirmation does not arrive within 5 seconds; gray Timeout badge remains until manually dismissed ✅

**Key Decisions:**
- `Confirmation` imported as `lapin::Confirmation` (public re-export), not `lapin::publisher_confirm::Confirmation` (private module)
- `mandatory=true` set unconditionally on every basic_publish call — defaulting to mandatory behavior for dev tools
- Timeout outcome is `Ok(PublishOutcome { status: 'timeout' })`, not `Err` — delivery outcome is not a command error
- `useRef` dismiss timer for auto-dismiss — avoids stale closure over badge state in setTimeout
- `shouldAdvanceTime: true` on `vi.useFakeTimers` — required for fake timer tests with async badge dismiss

**Success Criteria — all passed (human UAT 2026-05-19):**
1. User opens and closes the block library panel from a toggle button in the FormPanel header ✅
2. User creates a new block by entering a name and writing a JSON object in the CodeMirror editor; the block appears in the library list ✅
3. User edits an existing block's name or JSON content and saves it; the updated block is reflected immediately ✅
4. User deletes a block after confirming a prompt; the block is removed from the library ✅

### Phase 12: Block Library — Drag-and-Drop Layer

**Goal**: Users can drag a block from the library panel onto the form to merge its values into empty fields, with a warning when block keys have no matching form field
**Depends on**: Phase 11
**Requirements**: BLK-06, BLK-07, BLK-08
**Plans**: 3 plans

Plans:
**Wave 1** *(parallel — no file overlap)*
- [x] 12-01-PLAN.md — ProtoFormRenderer: applyBlockRef prop + useEffect wiring, field eligibility check, dirtyFields guard, TDD
- [x] 12-03-PLAN.md — BlockLibraryPanel: draggable="true" + onDragStart (blockId payload) + cursor-grab on block list rows, tests

**Wave 2** *(blocked on Wave 1 completion)*
- [x] 12-02-PLAN.md — FormPanel: drop zone on ScrollArea (onDragOver/onDragLeave/onDrop), isDraggingOver state, ring feedback, BLK-08 toast, tests

**Success Criteria — all passed (human UAT 2026-05-20):**
1. User drags a block card from the block library panel and drops it onto the form; fields in the form that are empty are populated with matching values from the block ✅
2. User drags a block onto a form where some fields have already been edited; only unmodified (not-dirty) fields are filled — no field the user touched is overwritten ✅
3. User drops a block whose keys include fields not present in the current form schema; a warning toast appears listing the skipped field names ✅
4. Drop zone absent in JSON mode — no ring appears, no form mutation occurs when in CodeMirror editor mode ✅

**Key Decisions:**
- dnd-kit (PointerSensor) replaces HTML5 DnD — WKWebView (macOS Tauri) `dataTransfer` API is broken
- `applyBlockRef` uses `dirtyFields` semantics per D-03 — a field typed back to default is not protected
- `setValue` without `shouldDirty: true` — block-filled fields remain non-dirty so subsequent drops can overwrite
- Block ID only in drag payload (block content looked up from store on drop, not transferred via DnD)
- dnd-kit `DndContext` + `DragOverlay` in AppLayout; `useDraggable` in BlockLibraryPanel; `useDroppable` in FormPanel scroll area
