## Overview

Plan Runner adds a full sequential test-plan workflow to Tap: define named plans with ordered steps, author each step with its own `.proto` file + target + response mode, execute the plan end-to-end, and monitor decoded replies inline and in a shared feed. No code required.

## Milestone Summary

**Decimal Phases:** None — all phases sequential (19–23).

**Key Decisions:**

- Isolated `StepFieldEditor` form instance per step (not shared with `ProtoFormRenderer`) — avoids frozen switch body; each step gets a clean form state on mount
- `usePlanExecutionStore` separate from `usePlanStore` — execution state is ephemeral per-run; persisting it would add complexity with no benefit
- Persistent AMQP connection per plan run (one connection, sequential steps) — intentional deviation from "ephemeral connection per operation" Key Decision; sequential steps on the same connection is simpler and avoids reconnect overhead
- Consumer-before-publish ordering for correlation-id mode — prevents race where reply arrives before consumer opens
- `forceMount` + CSS `hidden` on Editor TabsContent — preserves `StepFieldEditor` form state when user switches to Reply Feed tab without remounting

**Issues Resolved:**

- CR-01: IPC field name `step_id` (Rust snake_case) vs `stepId` (TS camelCase) mismatch — fixed in 22-05
- CR-02: Inverted `break` condition in cancel loop — would have caused runner to continue after cancel — fixed in 22-05
- Stale Radix UI state: `StepFieldEditor` not remounting on step switch — fixed by adding `key={selectedStepId}` to force remount
- `ScrollArea` height chain broken by missing `flex flex-col` on `TabsContent` — fixed during live UAT
- AMQP `correlation_id` property not propagating correctly — fixed in post-execution debug session

**Issues Deferred:**

- Pause / Resume mid-run — not selected for v1.6; deferred to v2
- Variable extraction and chaining across steps — depends on scripting engine; deferred to v2
- Phase 13 live-broker UAT items (8) — carried forward from v1.4; require running RabbitMQ instance

**Technical Debt Incurred:**

- `StepFieldEditor` uses a `useEffect` to watch `selectedStepId` changes for reset; this creates a two-render cycle on step switch. Acceptable for a dev tool, but could be refactored to ref-based reset.
- Reply feed timestamp is frontend-assigned (JS `Date.now()`) not AMQP delivery timestamp — AMQP timestamp field is optional and often unset by brokers.
