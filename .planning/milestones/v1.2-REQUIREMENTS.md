## v1.2 Requirements

### Bytes Field

- [x] **BFLD-01**: User can enter a bytes field value as a standard base64 string (RFC 4648, `+`/`/` alphabet) — **DELIVERED** Phase 6
- [x] **BFLD-02**: User can click a "From text" helper button to encode UTF-8 text into base64 in one action — **DELIVERED** Phase 6
- [x] **BFLD-03**: User sees an inline validation error when the input contains non-standard base64 characters (e.g. URL-safe `-`/`_`) — **DELIVERED** Phase 6
- [x] **BFLD-04**: User sees an error on send when the base64 value cannot be decoded — not silent empty bytes — **DELIVERED** Phase 6

### Map Field

- [x] **MFLD-01**: User can add and remove key-value row entries for a `map<K, V>` field — **DELIVERED** Phase 7
- [x] **MFLD-02**: User sees the key input constrained to the declared proto key type (numeric input for int keys, text for string, etc.) — **DELIVERED** Phase 7
- [x] **MFLD-03**: User sees an inline duplicate-key error and cannot send until all keys are unique — **DELIVERED** Phase 7 + quick task 260519-q01 (regression fix)
- [x] **MFLD-04**: User can fill in map values using the same field renderers as the rest of the form (scalar, enum, nested message) — **DELIVERED** Phase 7 (human UAT confirmed)
- [x] **MFLD-05**: User's map field encodes correctly as binary protobuf wire format when sent — **DELIVERED** Phase 7

### JSON Override

- [x] **JSON-01**: User can toggle between form view and raw JSON edit mode via a button in the form header — **DELIVERED** Phase 8
- [x] **JSON-02**: Switching to JSON mode pre-fills the editor with current form values (point-in-time snapshot, not a live feed) — **DELIVERED** Phase 8
- [x] **JSON-03**: Switching back to form mode applies the edited JSON to all form fields including repeated/map rows — **DELIVERED** Phase 8
- [x] **JSON-04**: User sees an error and explicit "Fix JSON / Discard" choice when switching back with invalid JSON — edits are never silently discarded — **DELIVERED** Phase 8
- [x] **JSON-05**: User sees a non-blocking warning listing unknown field names when the JSON contains keys not in the proto schema — **DELIVERED** Phase 8
- [x] **JSON-06**: JSON editor has syntax highlighting and dark mode support — **DELIVERED** Phase 8

## Out of Scope (unchanged)

| Feature | Reason |
|---------|--------|
| URL-safe base64 input mode | Adds ambiguity; standard alphabet is what prost-reflect produces — keep one alphabet |
| Persistent JSON-mode state across app restarts | JSON mode is a session-only power-user override; not worth persisting |
| Bidirectional canonical proto JSON ↔ form conversion | Canonical proto JSON (camelCase, oneof by field name) differs from form-internal shape; round-trip via form shape is sufficient |
| Map value types that are maps or repeated | Proto3 prohibits nested maps and repeated map values by spec |
