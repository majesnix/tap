# Milestone v1.1: Dark Mode

**Status:** ✅ SHIPPED 2026-05-18
**Phases:** 5
**Total Plans:** 3

## Overview

Proto Sender v1.1 adds full dark mode support: OS-aware theme detection on startup, an in-app toggle cycling through system / light / dark modes, cross-restart persistence via tauri-plugin-store, and manual visual UAT confirming correct rendering across all UI surfaces.

**Core Value:** Send a real protobuf message to RabbitMQ in under 30 seconds from a raw `.proto` file — no code, no curl, no manual encoding.

## Phases

### Phase 5: Dark Mode

**Goal:** Users can choose their preferred theme (system, light, or dark), have it applied immediately across the entire app, and find their choice remembered on next launch.
**Depends on:** Phase 4 (brownfield — all existing UI surfaces must be verified in dark mode)
**Plans:** 3 plans

Plans:
- [x] 05-01: ThemeProvider + ThemeBootstrap — next-themes wrapping App root with tauri-plugin-store persistence bridge and race-guard via bootstrapped flag (DRK-01, DRK-03)
- [x] 05-02: ThemeToggle component — icon cycle button (Monitor/Sun/Moon) in sidebar footer, wired to next-themes setTheme() with mounted guard (DRK-02)
- [x] 05-03: DRK-04 manual visual UAT checkpoint — human walkthrough of all UI surfaces in dark mode, approved

**Requirements delivered:** DRK-01, DRK-02, DRK-03, DRK-04

**Success Criteria verified:**
1. On first launch with no saved preference, the app matches the OS dark/light setting automatically
2. User can open a theme toggle in the app UI and switch between system, light, and dark modes — the change takes effect immediately without a reload
3. After switching theme and restarting the app, the previously selected mode is restored
4. Every existing UI surface — form panel, connection sidebar, publish bar, AMQP properties sheet, message history panel, response tab, modals, and shadcn/ui components — renders without visual defects in dark mode

## Milestone Summary

**Key Decisions:**

- next-themes ThemeProvider with `attribute="class"`, `defaultTheme="system"`, `enableSystem` chosen for OS preference detection — avoids custom matchMedia + CSS var management
- ThemeBootstrap child-of-ThemeProvider pattern: reads tauri-plugin-store on mount, gates mirror writes on `bootstrapped` boolean flag — prevents Pitfall 6 race where stale localStorage clobbers the store before bootstrap read completes
- `eslint-disable-line react-hooks/exhaustive-deps` on bootstrap useEffect is intentional — one-shot mount read, not reactive
- `CYCLE_ORDER: ["system", "light", "dark"]` array drives stateless mode progression — no state machine needed
- `aria-label` and `title` describe CURRENT mode (not next) per UI-SPEC copywriting contract
- Mounted guard returns disabled same-size Button placeholder (not null) to prevent layout shift during hydration
- DRK-04 verified by manual visual UAT only (D-07) — no automated snapshot or visual regression check substitutes for visual inspection

**Issues Resolved:**

- Bootstrap-mirror race condition (Pitfall 6) — solved via `bootstrapped` boolean flag gating the mirror effect
- Layout shift on ThemeToggle initial render — solved via disabled Button placeholder on pre-mount
- SSR/hydration safety for `useTheme()` — solved via `mounted` state guard

**Issues Deferred:**

- None

**Technical Debt Incurred:**

- `eslint-disable-line` on ThemeBootstrap mount useEffect (intentional, documented in summary)
- No E2E test for cross-restart persistence (requires full Tauri app + tauri-plugin-store integration — manual UAT is the check)
