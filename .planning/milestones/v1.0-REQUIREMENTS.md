# Requirements Archive: Proto Sender v1.0

**Archived:** 2026-05-18
**Milestone:** v1.0 MVP
**Core Value:** Send a real protobuf message to RabbitMQ in under 30 seconds from a raw `.proto` file — no code, no curl, no manual encoding.

> This file is a read-only archive of the v1.0 requirements. For active requirements, see the current REQUIREMENTS.md (or the next milestone's equivalent).

---

## v1 Requirements — All Delivered

### Connection Management

- [x] **CONN-01**: User can create and save named connection profiles (host, port, vhost, username, password, management API port) — ✓ Delivered Phase 2 (02-01)
- [x] **CONN-02**: User can switch between saved connection profiles with a single click — ✓ Delivered Phase 2 (02-02)
- [x] **CONN-03**: App tests connection reachability and credential validity when the user saves a profile — ✓ Delivered Phase 2 (02-02)
- [x] **CONN-04**: Passwords are stored in the OS keychain (macOS Keychain, Windows Credential Manager, Linux libsecret) — never in plain config files — ✓ Delivered Phase 2 (02-01)

### Proto Loading

- [x] **PROT-01**: User can open a `.proto` file via a file picker dialog at runtime (no pre-compilation step) — ✓ Delivered Phase 1 (01-01)
- [x] **PROT-02**: User can configure include paths (equivalent to `protoc -I`) so relative imports resolve correctly across project directory trees — ✓ Delivered Phase 1 (01-05)
- [x] **PROT-03**: Tool renders WellKnownTypes (google.protobuf.Timestamp, Duration, Any, etc.) with purpose-built form controls — ✓ Delivered Phase 3 (03-01)
- [x] **PROT-04**: User can have multiple `.proto` files open simultaneously and switch between message types within one session — ✓ Delivered Phase 3 (03-01)

### Form Editor — Field Types

- [x] **FORM-01**: Form renders scalar fields (string, int32, int64, uint32, uint64, sint32, sint64, float, double, bool) with appropriate input types and constraints — ✓ Delivered Phase 1 (01-01, 01-02)
- [x] **FORM-02**: Form renders nested message fields as expandable inline sub-forms — ✓ Delivered Phase 1 (01-03)
- [x] **FORM-03**: Form renders repeated fields as a list with add/remove item controls, including repeated nested messages — ✓ Delivered Phase 1 (01-03)
- [x] **FORM-04**: Form renders enum fields as dropdowns showing value names (not raw integer values) — ✓ Delivered Phase 1 (01-04)
- [x] **FORM-05**: Form renders oneof fields as a radio group; selecting a branch clears all sibling branches (correct proto wire semantics) — ✓ Delivered Phase 1 (01-04)

### Form Editor — Quality

- [x] **FORM-06**: App validates field values before send and surfaces errors inline — ✓ Delivered Phase 1 (01-02)
- [x] **FORM-07**: Form pre-populates sensible zero-value defaults on load (0 for numerics, empty string, first enum value) — ✓ Delivered Phase 1 (01-02)
- [x] **FORM-08**: App caps nested message expansion at 5 levels deep and shows a collapse placeholder below that — ✓ Delivered Phase 1 (01-03)
- [x] **FORM-09**: WellKnownType fields use purpose-built controls rather than raw field editors — ✓ Delivered Phase 1 (01-05)

### Publishing

- [x] **PUBL-01**: User can publish a message directly to a named queue (via the default exchange) — ✓ Delivered Phase 2 (02-04)
- [x] **PUBL-02**: User can publish a message to a named exchange with a user-specified routing key — ✓ Delivered Phase 2 (02-04)
- [x] **PUBL-03**: User can select target queues and exchanges from a live dropdown populated from the RabbitMQ Management API; falls back to manual text input when unavailable — ✓ Delivered Phase 2 (02-03)
- [x] **PUBL-04**: User can set AMQP message properties before sending (content-type, delivery-mode, TTL, correlation-id, reply-to, custom headers as key-value pairs) — ✓ Delivered Phase 3 (03-02)

### Message History

- [x] **HIST-01**: App logs all sent messages (timestamp, queue or exchange/routing-key, message type name, send status) — ✓ Delivered Phase 3 (03-03)
- [x] **HIST-02**: User can click any history entry to re-populate the form with its original field values and resend — ✓ Delivered Phase 3 (03-04)
- [x] **HIST-03**: User can view the binary payload of any history entry as a hex string — ✓ Delivered Phase 3 (03-03)
- [x] **HIST-04**: User can filter the history log by message type name or by queue/exchange name — ✓ Delivered Phase 3 (03-04)

### Reply Queue Consumer

- [x] **RESP-01**: User can select a reply/response queue to read from — live dropdown from the Management API, falling back to manual text input when unavailable — ✓ Delivered Phase 4 (04-02)
- [x] **RESP-02**: App reads (consumes) one incoming message from the selected reply queue on a "Read" button click; shows a clear "Queue empty" state when no messages are waiting — ✓ Delivered Phase 4 (04-01)
- [x] **RESP-03**: App deserializes the binary protobuf payload using the currently loaded proto schema and selected message type, displaying all field values in a structured read-only view — ✓ Delivered Phase 4 (04-01)
- [x] **RESP-04**: App acknowledges (basic.ack) the consumed message after successful deserialization, permanently removing it from the queue — ✓ Delivered Phase 4 (04-01)
- [x] **RESP-05**: Decoded response view shows both the human-readable field values and the raw hex payload side by side — ✓ Delivered Phase 4 (04-01, 04-02)

## Out of Scope

| Feature | Reason |
|---------|--------|
| Real-time queue monitoring / stream inspection | Different product, not core to the send-test loop |
| Team cloud sync / shared profiles | Each user manages local profiles; out of scope for v1 |
| OAuth or federated credential management | Email/password RabbitMQ auth sufficient for dev tool |
| Non-proto message formats (JSON-only, Avro, etc.) | Binary protobuf only in v1 |
| Request scripting / automation | Postman-style scripting adds scope without core value |
