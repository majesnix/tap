Create src/hooks/usePlanRunner.ts:

Pattern analog: src/components/response/SubscribePanel.tsx — follow its pattern of
encapsulating IPC lifecycle (start/stop actions, reactive state from Zustand store,
in-flight cancellation guard). The runner hook is the subscription analog for plan
execution.

The hook accepts no props. It reads state and actions from usePlanExecutionStore.
It reads the active profile from useConnectionStore (existing store — check what
selector gives the active profile name; likely activeProfile or similar).

Exports:
- startRun(plan: Plan): Promise<void>
- stopRun(): Promise<void>
- isRunning: boolean (derived from store.runningPlanId !== null)

startRun implementation:
1. Call setRunning(plan.id, plan.steps.map(s => s.id)) — this initializes ALL steps
   to 'pending' immediately so Pending badges appear before any step executes (D-14)
2. Initialize local succeeded = 0, total = plan.steps.length
3. Determine stop_on_error: plan.stop_on_error ?? true (D-07)
4. For each step of plan.steps (for...of loop — sequential):
   a. setActiveStep(step.id)
   b. setStepStatus(step.id, 'sending')
   c. If step.response_mode.mode !== 'no-wait': setStepStatus(step.id, 'waiting-response')
      after a short yield (or set it right before the invoke call)
      Actually: set 'sending' right before invoke; the invoke itself handles the wait.
      Set 'waiting-response' simultaneously with 'sending' for reply modes — or set
      'sending' first then immediately 'waiting-response' before the await. Follow
      whichever is clearer. The UI distinction is: 'sending' for no-wait, 'waiting-response'
      for reply modes (both are amber in the badge).
   d. try/catch around: const result = await executeStep(activeProfile, step)
   e. On success (result.status === 'done'):
      - setStepStatus(step.id, 'done')
      - succeeded++
   f. On error (result.status === 'error' or catch):
      - setStepStatus(step.id, 'error')
      - toast.error('Step \'' + step.name + '\' failed: ' + (result.error ?? 'Unknown error'))
        (import toast from sonner)
      - if stop_on_error AND !store.isCancelling: break
        (pitfall #8: if isCancelling, don't break — the loop will exit naturally)
5. setActiveStep(null)
6. setSummary(succeeded, total)
7. Call finishRun() — sets runningPlanId=null and activeStepId=null while keeping
   stepStatuses and summary visible for the UI to display post-run state.
   Do NOT call clearRun() here — that would erase the badges and summary line.
   clearRun() is called implicitly by setRunning() at the START of the next run.

stopRun implementation:
1. setIsCancelling(true)
2. await cancelPlanRun()
   (The in-flight execute_step will return with status 'error', error 'Cancelled'.
   The pitfall #8 check in startRun handles this gracefully.)

Note: Task 2 adds finishRun() usage to usePlanExecutionStore.ts (which was defined
in Task 1). If finishRun() was not added in Task 1, add it to the store now before
implementing the hook. The store file is listed in this task's files for this reason.
