## Gap Detail: MFLD-03 (BLOCKER)

**Requirement:** "User sees an inline duplicate-key error and the send button is blocked until all keys in the map are unique"

**Current behavior:**
- ✓ Inline duplicate-key error is shown (setError renders visual error in MapField.tsx)
- ✗ Send button is NOT disabled when duplicate keys exist

**Root cause — two independent failures:**

1. **RHF gate failure:** Phase 7 REVIEW-FIX IN-04 removed the `hasDuplicatesRef` + hidden `Controller` with `validate` rule. The replacement (`setError` on `__mapDuplicateGuard` in a `useEffect`) does not reliably keep `formState.isValid = false` — RHF's own onChange pass clears manually-set errors when no `validate` rule is registered for the field.
   - **Evidence:** Test `MapField > formState.isValid is false while duplicates exist` FAILS (1 failure in every phase run post-Phase-7)

2. **PublishBar wiring missing:** `PublishBar.tsx:128` `canSend = isConnected && hasTarget` — `formState.isValid` is never consulted. So even if the RHF gate were fixed, the button would still not be disabled.
   - **Evidence:** `grep "isValid\|formState" src/components/publish/PublishBar.tsx` → 0 matches

**Fallback present:** `encode.rs` WR-02 (commit `1329f45`) rejects duplicate keys at Rust encode layer with `AppError::EncodeError("duplicate map key")`, surfaced as a toast error. This is a last-resort guard — the button is not disabled, but an erroneous send with duplicate keys does fail visibly.

**Fix required:**
```
Option A — Restore hidden Controller:
  Re-add the hidden Controller (guardName) with a validate rule that reads hasDuplicatesRef.current
  (the Code Reviewer's IN-04 finding was incorrect — the Controller IS needed for reliable isValid gate)
  + wire formState.isValid into PublishBar canSend

Option B — Use a Zustand signal:
  Add a mapHasDuplicates flag to the form store; MapField writes it; PublishBar reads it
  (avoids RHF internals, more explicit)
```
