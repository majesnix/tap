# Quick Task 260519-q01: Fix MFLD-03 Send-Button Blocking on Duplicate Map Keys

## What Was Changed

### Task 1 — MapField.tsx

**File:** `src/components/form/fields/MapField.tsx`

Replaced the `setError`/`clearErrors` `useEffect` (introduced by IN-04 review-fix) with the `register`+`unregister`+`trigger`+`setValue` approach described in the plan.

Changes:
- Added `useRef` to React import
- Replaced `setError, clearErrors` with `register, unregister, trigger, setValue` in `useFormContext` destructure
- Added `hasDuplicatesRef = useRef(false)` after `hasDuplicates` line; assigned `hasDuplicatesRef.current = hasDuplicates` synchronously during render so the validate closure is never stale
- Replaced the single `setError/clearErrors` `useEffect` with TWO effects:
  1. Register effect (`[register, unregister, guardName]`): registers the guard field with a `validate` rule that reads `hasDuplicatesRef.current`
  2. Trigger effect (`[hasDuplicates, guardName, setValue, trigger]`): calls `setValue(guardName, hasDuplicates)` + `void trigger(guardName)` whenever duplicate state changes — `setValue` ensures RHF tracks the field as "touched" so `isValid` is properly updated
- Updated JSDoc comment to describe the new mechanism

**Why register+trigger vs setError:** In RHF `mode: "onChange"`, `setError` on an unregistered field does not reliably affect `formState.isValid`. The `register(name, { validate })` + `trigger(name)` pattern is RHF-idiomatic: the validate rule runs on trigger and its result is included in `isValid` computation. Using `setValue(guardName, hasDuplicates)` before `trigger` ensures RHF includes the field in the validation pass.

### Task 2 — PublishBar.tsx

**File:** `src/components/publish/PublishBar.tsx`

Two-line change:
- Added `encodeError` to the `useProtoStore` destructure
- Updated `canSend = isConnected && hasTarget && !encodeError`

This ensures the Send button is also blocked when `encodeMessage` fails due to duplicate map keys (WR-02 `AppError::EncodeError` path), providing a second layer of protection beyond the form validity gate.

## Deviations from Plan

**Minor:** The plan described a two-effect approach using only `register`+`trigger` (without `setValue`). During implementation, a debug investigation revealed that `trigger(guardName)` alone was insufficient to update `formState.isValid` in the JSDOM/vitest test environment — the field needed `setValue(guardName, hasDuplicates)` first to ensure RHF marked the field as "touched" before running validation. Added `setValue` to the trigger effect. This is consistent with the plan's intent and does not change the external behavior.
