# Quick Task 260519-q01: Fix MFLD-03 Send-Button Blocking on Duplicate Map Keys

## Context

MFLD-03 requirement: "User sees an inline duplicate-key error and the send button is blocked until all keys in the map are unique."

### What broke

Phase 7 REVIEW-FIX IN-04 removed the hidden `Controller` with `validate` rule, replacing it with `setError`/`clearErrors` in a `useEffect`. The reviewer's claim was "RHF only runs Controller validate rules when the field's own value changes, so the Controller adds no value." This was incorrect.

The `setError` replacement fails because:
- In RHF `mode: onChange`, a new onChange event can re-calculate `isValid` from registered validators only
- An error on an unregistered field may or may not persist depending on RHF internals and timing
- Test `MapField > formState.isValid is false while duplicates exist` FAILS (confirmed in 08-01 + 08-02 SUMMARY)

Additionally, `PublishBar.canSend = isConnected && hasTarget` never checks form validity. Even if the RHF gate worked, the Send button would remain visually enabled.

### Two-layer fix

1. **MapField.tsx**: switch to `register(guardName, { validate })` + `trigger(guardName)` â€” this is RHF-idiomatic: register a virtual field with a validate rule, then force re-validation via `trigger` when the duplicate state changes. A `useRef` ensures the validate closure always reads the current `hasDuplicates` value without going stale.

2. **PublishBar.tsx**: add `!encodeError` to `canSend`. When duplicate keys cause `encodeMessage` (Rust WR-02 `map.contains_key` check) to return `AppError::EncodeError`, FormPanel's catch sets `setHexPreview("")` and `setEncodeError(msg)`. The encodeError signal is already in `useProtoStore` â€” PublishBar just needs to read it.


2. Update `useFormContext` destructure â€” replace `setError, clearErrors` with `register, unregister, trigger`:
   ```tsx
   const { control, register, unregister, trigger } = useFormContext();
   ```

3. After `const hasDuplicates = duplicateKeys.size > 0;` (line 130), add the ref:
   ```tsx
   const hasDuplicatesRef = useRef(false);
   hasDuplicatesRef.current = hasDuplicates;
   ```
   The assignment runs during render, before any effects fire â€” so `hasDuplicatesRef.current` is current when the trigger effect runs.

4. Replace the existing single `useEffect` (currently lines 137-143) with two effects:

   ```tsx
   // Register guard field with validate rule â€” keeps formState.isValid false while duplicates exist.
   useEffect(() => {
     register(guardName, {
       validate: () => (hasDuplicatesRef.current ? "Duplicate key" : true),
     });
     return () => unregister(guardName);
   }, [register, unregister, guardName]);

   // Force re-validation of the guard field whenever duplicate state changes.
   useEffect(() => {
     void trigger(guardName);
   }, [hasDuplicates, trigger, guardName]);
   ```

5. Update the JSDoc comment (lines 103-105) to describe the new mechanism:
   ```tsx
   * - register(guardName, { validate }) + trigger(guardName) keeps formState.isValid false while duplicates exist
   ```
   Remove the stale "No Controller is needed" comment (lines 134-136).

## Task 2: Fix PublishBar.tsx

**File:** `src/components/publish/PublishBar.tsx`

### Changes

1. Add `encodeError` to the `useProtoStore` destructure (currently line 81):
   ```tsx
   const { hexPreview, encodeError } = useProtoStore();
   ```

2. Update `canSend` (currently line 128):
   ```tsx
   const canSend = isConnected && hasTarget && !encodeError;
   ```

### Why this works

When duplicate map keys trigger `AppError::EncodeError` in Rust (WR-02 fix, commit `1329f45`), FormPanel's encode effect catches it and calls `setEncodeError(msg)`. This sets `useProtoStore.encodeError` to a non-null string. `!encodeError` in `canSend` then disables the Send button.

When the user resolves duplicates, the next debounce fires `encodeMessage` again, FormPanel calls `setEncodeError(null)` at the start of the attempt, and on success `encodeError` is cleared â†’ button re-enables.

### Note on existing test coverage

PublishBar.test.tsx uses `mockInvoke` for IPC. No new tests needed for this change â€” the encode error flow is already exercised by the encode path. The MapField test covers the `formState.isValid` gate.
